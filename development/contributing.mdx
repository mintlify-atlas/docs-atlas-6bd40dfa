## Getting Started

Thank you for your interest in contributing to Food For All DC! This guide will help you set up your development environment and understand our contribution workflow.

### Prerequisites

- **Node.js:** v16+ (recommended: v18+)
- **Python:** 3.11+ (for Cloud Functions)
- **Firebase CLI:** Latest version (optional)
- **Git:** Latest version

### Initial Setup

```bash
# Clone the repository
git clone <your-repo-url>
cd food-for-all-dc/my-app

# Install dependencies
npm install

# Start development server
npm start  # Opens at http://localhost:3000
```

### Firebase Setup (Optional)

For local testing with Firebase emulators:

```bash
# Install Firebase CLI
npm install -g firebase-tools
firebase login

# Setup Python environment for Cloud Functions
cd functions-python
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# Start Firebase emulators
cd ..
firebase emulators:start
```

<Warning>
Use the Firebase Emulator for local testing only. Never test against production Firebase directly.
</Warning>

### Environment Variables

Copy the example environment file and add your API keys:

```bash
cp .env.example .env
# Edit .env with your API keys
```

**Required Variables:**

```bash
REACT_APP_FIREBASE_API_KEY=your-api-key
REACT_APP_FIREBASE_AUTH_DOMAIN=your-auth-domain
REACT_APP_FIREBASE_PROJECT_ID=your-project-id
REACT_APP_FIREBASE_STORAGE_BUCKET=your-storage-bucket
REACT_APP_FIREBASE_MESSAGING_SENDER_ID=your-sender-id
REACT_APP_FIREBASE_APP_ID=your-app-id
REACT_APP_FIREBASE_MEASUREMENT_ID=your-measurement-id
REACT_APP_GOOGLE_MAPS_API_KEY=your-maps-key
```

<Info>
Never commit `.env` files to version control. They're included in `.gitignore` by default.
</Info>

## Code Standards

### TypeScript Guidelines

**DO:**
- Use TypeScript for all new files
- Prefer `interface` over `type` for object shapes
- Import types from `src/types/` when available
- Use explicit return types for functions

**DON'T:**
- Avoid `any` type - use proper type definitions
- Don't use `@ts-ignore` without explanation

```typescript
// ✅ Good
import { ClientProfile } from '../types';

const getClient = async (uid: string): Promise<ClientProfile> => {
  // Implementation
};

// ❌ Bad
const getClient = async (uid: any): Promise<any> => {
  // Implementation
};
```

### Component Guidelines

**Use functional components with hooks:**

```typescript
import React, { useState, useEffect } from 'react';

const MyComponent: React.FC = () => {
  const [data, setData] = useState<string>('');

  useEffect(() => {
    // Effect logic
  }, []);

  return <div>{data}</div>;
};

export default MyComponent;
```

**Keep components focused and reusable:**
- Single responsibility per component
- Extract business logic into services or utilities
- Use common components from `src/components/common/`

### Styling Guidelines

**Use CSS variables from `src/styles/theme.css`:**

```css
/* ComponentName.module.css */
.container {
  padding: var(--spacing-md);
  background-color: var(--color-background-main);
}

.title {
  color: var(--color-text-primary);
  font-size: var(--font-size-lg);
}
```

**Use CSS Modules for component styles:**

```typescript
import styles from './ComponentName.module.css';

<div className={styles.container}>
  <h2 className={styles.title}>Title</h2>
</div>
```

**Use common components when possible:**

```typescript
import { Button, Input, Modal } from '../components/common';

<Button variant="primary" size="small">Save</Button>
<Input label="Name" value={name} onChange={setName} />
```

See the [Styling Guide](/development/styling-guide) for detailed styling guidelines.

## Architecture Patterns

### API Keys & Configuration

**All API keys are centralized in `src/config/apiKeys.ts`:**

```typescript
import { googleMapsApiKey, firebaseConfig } from '../config/apiKeys';

// Use in your code
const map = initializeMap(googleMapsApiKey);
```

**Adding a new API key:**

1. Add to `.env.example`:
   ```bash
   REACT_APP_NEW_API_KEY=placeholder
   ```

2. Add to `src/config/apiKeys.ts`:
   ```typescript
   export const newApiKey = getRequiredEnv('REACT_APP_NEW_API_KEY');
   // or getOptionalEnv() for optional keys
   ```

3. Import and use:
   ```typescript
   import { newApiKey } from '../config/apiKeys';
   ```

### Service Pattern

**All services use the singleton pattern:**

```typescript
class MyService {
  private static instance: MyService;

  private constructor() {}

  static getInstance(): MyService {
    if (!MyService.instance) {
      MyService.instance = new MyService();
    }
    return MyService.instance;
  }

  async doSomething(): Promise<void> {
    try {
      // Implementation
    } catch (error) {
      throw formatServiceError(error, "Operation failed");
    }
  }
}

export default MyService;
```

**Usage:**

```typescript
const service = MyService.getInstance();
await service.doSomething();
```

**Error handling:**

```typescript
import { formatServiceError } from '../utils/serviceError';
import { useNotification } from '../context/NotificationProvider';

try {
  await service.doSomething();
} catch (error) {
  const formatted = formatServiceError(error, "Failed to complete action");
  showNotification(formatted.message, "error");
}
```

### State Management

**Global State (React Context):**
- `AuthProvider` - User authentication and roles
- `ClientDataContext` - Client data caching
- `RecurringDeliveryContext` - Recurring deliveries

**Local State:**
- Component state for forms and UI
- Always refresh context after mutations

```typescript
import { useClientData } from '../context/ClientDataContext';

const MyComponent = () => {
  const { clients, refreshClients } = useClientData();

  const handleUpdate = async () => {
    await clientService.updateClient(data);
    await refreshClients(); // Refresh context
  };
};
```

### Routing & Protection

**Routes are defined in `src/routesConfig.tsx`:**

```typescript
import ProtectedRoute from './auth/ProtectedRoute';
import { UserType } from './types';

<Routes>
  <Route element={<ProtectedRoute allowedRoles={[UserType.Admin, UserType.Manager]} />}>
    <Route path="/admin" element={<AdminPage />} />
  </Route>
</Routes>
```

**Available Roles:**
- `UserType.Admin` - Full access
- `UserType.Manager` - Client and delivery management
- `UserType.ClientIntake` - Client profile editing
- `UserType.Driver` - View-only delivery access

**Lazy load page components:**

```typescript
const Calendar = lazy(() => import('./pages/Calendar/Calendar'));
```

## Naming Conventions

### Files
- **Components/Types/Enums:** `PascalCase` (e.g., `ClientProfile.tsx`)
- **Variables/Functions:** `camelCase`
- **Constants:** `UPPER_SNAKE_CASE`

### Code
- **Booleans:** `is/has/should` prefix (e.g., `isLoading`, `hasError`)
- **Event Handlers:** `handle/on` prefix (e.g., `handleClick`, `onSubmit`)
- **Custom Hooks:** `use` prefix (e.g., `useAuth`, `useClientData`)
- **Services:** `Service` suffix (e.g., `ClientService`)

## Pull Request Process

### Before Submitting

**1. Run checks:**

```bash
npm run lint       # Must pass
npm run lint:fix   # Auto-fix issues
npm run build      # Must succeed
npx tsc --noEmit   # TypeScript check
```

**2. Code review checklist:**

- [ ] Use `NotificationProvider` for errors (no `console.log`)
- [ ] Use `src/config/apiKeys.ts` for API keys
- [ ] Use CSS variables from `src/styles/theme.css`
- [ ] Use common components when appropriate
- [ ] Error handling uses `formatServiceError()`
- [ ] Services use singleton pattern
- [ ] No `any` types
- [ ] TypeScript strict mode passes

**3. Test your changes:**

- Test with different user roles if applicable
- Test edge cases and error scenarios
- Verify responsive design on different screen sizes

### PR Title Format

Use conventional commit format:

- `Feature: [description]` - New features
- `Fix: [description]` - Bug fixes
- `Refactor: [description]` - Code improvements
- `Docs: [description]` - Documentation
- `Style: [description]` - Styling changes

**Examples:**
- `Feature: Add client export to CSV`
- `Fix: Resolve calendar date picker issue`
- `Refactor: Extract delivery logic to service`

### PR Description Template

```markdown
## What
Brief description of what this PR does.

## Why
Why is this change needed?

## Testing
How did you test this?
- [ ] Tested with Admin role
- [ ] Tested with Manager role
- [ ] Tested error scenarios
- [ ] Tested on mobile

## Screenshots
(If UI changes)
```

## Common Workflows

### Adding a New Page

1. **Create component:**
   ```bash
   src/pages/NewPage/NewPage.tsx
   src/pages/NewPage/NewPage.module.css
   ```

2. **Add route** in `src/routesConfig.tsx`:
   ```typescript
   const NewPage = lazy(() => import('./pages/NewPage/NewPage'));

   <Route element={<ProtectedRoute allowedRoles={[UserType.Admin]} />}>
     <Route path="/new-page" element={<NewPage />} />
   </Route>
   ```

3. **Add navigation link** in `src/pages/Base/Base.tsx` (if needed)

4. **Use context hooks** (`useAuth`, `useClientData`) and services

### Modifying Client Data

1. **Update type** in `src/types/client-types.ts`
2. **Update form components** (`src/components/ClientProfile.tsx`, `src/pages/Profile/`)
3. **Add validation** to `src/utils/firestoreValidation.ts`
4. **Update Spreadsheet** in `src/components/Spreadsheet/` if visible/editable
5. **Update export** in `src/config/exportConfig.ts` if exported
6. **Update `ClientService`** methods if needed

### Firebase Operations

**Always validate before writes:**

```typescript
import { validateClientProfile } from '../utils/firestoreValidation';

const errors = validateClientProfile(clientData);
if (errors.length > 0) {
  throw new Error(`Validation failed: ${errors.join(', ')}`);
}

await setDoc(docRef, clientData);
```

**Wrap errors with `formatServiceError()`:**

```typescript
try {
  await setDoc(docRef, data);
} catch (error) {
  throw formatServiceError(error, "Failed to save client");
}
```

**Show errors via `NotificationProvider`:**

```typescript
import { useNotification } from '../context/NotificationProvider';

const { showNotification } = useNotification();

try {
  await saveClient(data);
  showNotification("Client saved successfully", "success");
} catch (error) {
  showNotification(error.message, "error");
}
```

**Firestore Collections:**
- `client-profile2` - Client profiles
- `events` - Delivery events
- `users` - User accounts
- `Drivers2` - Driver information
- `clusters` - Route clusters
- `referral` - Referral agencies
- `limits` - System limits
- `tags` - Client tags

### Performance Optimization

**Lazy load page components:**

```typescript
const Calendar = lazy(() => import('./pages/Calendar/Calendar'));
```

**Use `React.memo()` for expensive components:**

```typescript
const ExpensiveComponent = React.memo(({ data }) => {
  // Component logic
});
```

**Use `react-virtuoso` for large lists:**

```typescript
import { Virtuoso } from 'react-virtuoso';

<Virtuoso
  data={largeDataset}
  itemContent={(index, item) => <Item data={item} />}
/>
```

**Cache with context (5 min TTL):**
- Roles cached in `AuthProvider`
- Recurring deliveries cached in `RecurringDeliveryContext`

## Git Workflow

### Branch Strategy

1. **Create feature branch from `main`:**
   ```bash
   git checkout main
   git pull origin main
   git checkout -b feature/my-new-feature
   ```

2. **Make changes and commit:**
   ```bash
   git add .
   git commit -m "Feature: Add client export functionality"
   ```

3. **Push and create PR:**
   ```bash
   git push origin feature/my-new-feature
   ```

4. **After merge, delete branch:**
   ```bash
   git checkout main
   git pull origin main
   git branch -d feature/my-new-feature
   ```

### Commit Messages

Use conventional commit format:

```bash
Feature: Add client search functionality
Fix: Resolve calendar rendering issue
Refactor: Extract auth logic to service
Docs: Update README with setup instructions
```
